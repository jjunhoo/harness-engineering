# 자동 테스트 생성

에이전트가 **구현과 동시에 또는 직후에** 테스트를 생성·갱신하는 절차를 정의한다. 목적은 회귀 방지와 `AGENTS.md` [5] 유닛·통합 층의 **증거 자동화**다.

## 1. 범위

| 유형 | 설명 | 우선순위 |
|------|------|-----------|
| 유닛/컴포넌트 | 공개 API·순수 함수·도메인 규칙 | 필수 |
| 계약 | 입출력 스키마·에러 코드 | 변경 시 |
| 통합 | DB·HTTP 목(stub) 기반 | 리스크 높을 때 |
| E2E | `evaluations/scenarios/`에 링크되는 자동 스크립트 | 핵심 사용자 여정 |

## 2. 생성 원칙

1. **`plans/`에 검증 전략**이 있어야 한다(무엇을 깨뜨리면 안 되는지).
2. **실패 우선**: 해피 패스만이 아니라 경계·에러 코드를 포함한다.
3. **결정적**: 플레이크는 `RELIABILITY.md`에 따라 격리·재시도 정책을 문서화한다.
4. **레이어 준수**: `Types`/`Service` 등 상위에서 I/O 없는 테스트를 우선한다(`architecture/LAYERED_MODEL.md`).
5. **인간 검토 없이 머지 금지 조항 없음**: 자동 생성은 **PR의 일부**로 리뷰·CI를 통과해야 한다.

## 3. 워크플로 삽입 지점

- `docs/WORKFLOW.md` **3–4단계** 사이: 구현 직후 테스트 초안 생성 → 로컬 실행 → 실패 시 구현 수정.
- 멀티 리뷰에서 QA는 **테스트가 요구사항을 반영하는지**를 본다.

## 4. 산출물

- 테스트 소스 파일 + (선택) 스냅샷·픽스처
- PR 본문 `How verified`에 실행 명령
- 회귀 위험이 있으면 `evaluations/scenarios/` 갱신
