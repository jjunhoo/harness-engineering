# 피드백 시스템 (다층)

`AGENTS.md` [5]에 따라 피드백을 층으로 나누고, 각 층에 **지표·도구·에이전트 접근법**을 붙인다.

## 층 1: 유닛

- **지표**: 테스트 통과율, 실패 테스트 수, (선택) 커버리지
- **도구**: 언어별 테스트 러너
- **에이전트**: CI 로그, 로컬 동일 명령 재현

## 층 2: 통합

- **지표**: E2E 통과, 플레이크율, 픽스처 준비 시간
- **도구**: 계약 테스트, 컨테이너 기반 통합 스위트
- **에이전트**: `evaluations/scenarios/` 문서화된 시나리오 실행

## 층 3: 전체 시나리오 평가

- **지표**: 루브릭 점수(`QUALITY_SCORE.md`), 체크리스트 완료율
- **도구**: `evaluations/rubrics/`, `evaluations/checklists/`
- **에이전트**: PR에 루브릭 자기 채점 첨부(정직하게 실패 표기)

## 층 4: 운영 (로그·메트릭·트레이스)

- **지표**: 에러율, p95 지연, SLO 위반 건수
- **도구**: 관측 스택(팀 선택) — 중요한 것은 **에이전트가 읽을 수 있는 출력**
- **에이전트**: `references/TOOLS.md`에 “어디서 무엇을 읽는지” 고정

### 하네스 불변조건

> 에이전트가 로그, 메트릭, UI 상태를 직접 읽을 수 있어야 한다.

“사람만 대시보드에 접근 가능”은 **금지**다. 최소한 CI가 주기적으로 export하거나, 읽기 전용 API 키로 스냅샷을 저장소에 붙이는 패턴을 쓴다.

## 층 4b: 로그 기반 피드백(전용)

운영 층 4의 하위로, **구조화 로그를 입력으로 한 트리아지·집계·조치 제안**을 다룬다.

- **지표**: 이벤트율, `error.type`별 카운트, 동일 `correlation_id` 체인 길이
- **도구**: JSONL export, CI 아티팩트, (선택) 로그 쿼리 MCP
- **에이전트**: `docs/LOG_FEEDBACK.md`, `evaluations/feedback-logs/`, 시나리오 `evaluations/scenarios/LOG_BASED_TRIAGE.md`
- **출구**: 패치 PR 초안 또는 `docs/AUTO_IMPROVEMENT_ON_FAILURE.md` 루프
