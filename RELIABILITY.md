# 성능 및 안정성 규칙 (Reliability)

AI 주도 개발이 **운영 가능한 서비스**로 이어지도록 최소 기준을 정한다.

## 1. 관측성 (Observability)

- **로깅**: 구조화 로그(JSON 권장), 상관 ID, PII 최소화 (`SECURITY.md` 정합)
- **메트릭**: RED/USE 중 서비스 유형에 맞는 하위 집합은 필수로 정의
- **트레이스**: 분산 호출이 있으면 샘플링 정책을 문서화

에이전트는 `references/TOOLS.md`에 적힌 경로·도구로 로그/메트릭/트레이스를 **직접 읽을 수 있어야** 한다 (export, 읽기 전용 API, MCP).

## 2. SLO 사고 방식

- 핵심 사용자 여정마다 **목표 가용성·지연**을 한 줄이라도 적는다 (`docs/` 또는 서비스 README).
- 에러 버짓: 배포 단위로 허용 가능한 실패율을 정하고 초과 시 중단 규칙을 둔다.

## 3. 실패 모드

- **타임아웃**: 모든 외부 호출에 상한
- **재시도**: 멱등할 때만 지수 백오프
- **서킷 브레이커**: 연쇄 장애 방지
- **롤백**: 플래그 또는 배포 이전 버전으로 복귀 경로

## 4. 장시간 자율 실행 (6시간+)

- 작업을 **체크포인트**가 있는 단위로 분할 (`plans/`).
- 각 단계는 **멱등**하게 설계(재실행 안전).
- 중간 산출물을 저장소 또는 아티팩트에 기록해 중단 시 이어가기 가능하게 한다.

### 체크포인트 프로토콜 (에이전트)

| 체크포인트 | 기록 위치 | 최소 내용 |
|------------|-----------|-----------|
| 계획 동결 | `plans/` | 단계, 리스크, 검증 명령 |
| 구현 중간 | 동일 파일 또는 PR 코멘트 | 완료 단계·남은 단계·차단 이슈 |
| 테스트 완료 | PR 본문 | 통과한 테스트 식별자 |
| 머지 직전 | PR 또는 태그 | 롤밄 요약 |

## 5. 성능

- 성능 목표가 있으면 재현 가능한 벤치마크 또는 부하 시나리오를 `evaluations/scenarios/`에 둔다.
- 조기 최적화보다 **측정 기반** 최적화를 우선한다.
