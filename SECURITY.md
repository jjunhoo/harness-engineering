# 보안 정책 (Security)

하네스는 **비밀을 저장소에 두지 않는다**를 기본 불변조건으로 한다.

## 1. 비밀 관리

- API 키, 비밀번호, 개인 토큰, 클라우드 자격증명은 **환경 변수·시크릿 매니저·OIDC**로만 주입한다.
- 커밋 전 로컬 훅 또는 `git secrets`류 도구 사용을 권장한다.
- 유출 의심 시 즉시 회전(rotate)하고, `plans/`에 사건 요약(비밀 본문 없이)을 남긴다.

## 2. 공급망

- 의존성은 잠금 파일(lockfile)으로 고정한다.
- CI에서 검증 가능한 해시/서명 검증을 사용한다 (언어별 표준 도구).

## 3. 접근 제어

- 최소 권한: 사람·에이전트·CI 역할별 권한 분리
- 저장소 설정: 브랜치 보호, 필수 리뷰, 필수 상태 검사

## 4. 데이터

- PII는 목적에 필요한 최소한만, 보존 기간 명시
- 로그에 비밀·토큰·전체 카드번호 등이 남지 않도록 마스킹

## 5. 에이전트 특유 위험

- **프롬프트 인젝션** 가능 영역(플러그인, 외부 콘텐츠 처리)은 입력 검증·출력 이스케이프·권한 샌드박스를 고려한다.
- 외부 도구(MCP, 셸) 호출은 허용 목록과 감사 로그를 권장한다.

## 6. 취약점 보고

보고 채널과 SLA는 조직 정책에 따른다. 본 저장소는 공개 예시일 경우 Issue 대신 비공개 채널을 안내한다.

## 7. CI·품질 게이트와 연동된 차단 조건 (하드 게이트)

다음은 `QUALITY_SCORE.md` 하드 게이트와 합쳐져 **자동으로 머지를 막아야 하는** 예시다.

- 커밋에 API 키·장기 자격증명·프라이빗 키 블록이 포함됨
- 서드파티 패키지를 잠금 파일 없이 임의 버전으로 고정함(팀이 lockfile을 표준으로 둔 경우)
- 브랜치 보호 규칙을 우회하려는 변경(예: CI 필수 검사 제거) — 정당한 이유는 ADR + 승인

구체적 자동 탐지(시크릿 스캔, 정책-as-code)는 조직 도구에 맞춰 CI 단계로 추가한다.
